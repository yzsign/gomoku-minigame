# 对手战绩弹窗 · 添加好友 — 需求与技术方案

**版本**：V1.6  
**产品**：团团五子棋微信小游戏  
**涉及模块**：对局结束页、对手战绩弹窗（含随机匹配中点头像）、好友系统、用户级 WebSocket（本期不含独立「消息中心」页）

---

## 1. 需求背景

对局结束后可查看对手战绩弹窗，但仅展示战绩，无添加好友入口；随机匹配中点击对手头像亦可打开同一类「对手战绩」弹窗。玩家无法在赛后快速建立社交关系，影响互动与留存。

**目标**：在对手战绩弹窗内完成「发起 / 展示状态」的闭环；被申请人侧通过 **用户级 WebSocket + 弹框** 处理同意/拒绝（见 §4.3）。交互简单，不遮挡关闭、战绩信息、再来一局等原有操作。

---

## 2. 功能入口

| 场景 | 说明 |
|------|------|
| 对局结束页 | 打开对手战绩弹窗 |
| 随机匹配 / 联机对局中 | 点击对手头像 → `showOpponentRatingModal` 拉取对手公开战绩 |
| 我的战绩列表 → 点头像 | `showHistoryOpponentRatingModal`；**加好友本期不实现**，产品保留方案见 **§13** |

**已定**：**对局结束页**、**随机匹配/联机对局中点头像** 打开的「对手战绩」弹窗均实现「添加好友」能力。**历史战绩列表**点头像（`showHistoryOpponentRatingModal`）：**产品方案保留**与同弹窗一致的加好友能力（见 **§13**），**本期不开发**（见 **§10**）。

**UI 位置**：对手信息（昵称、胜率、积分等）下方，**居中**，「添加好友」按钮。

**规格（PRD）**：

- 按钮高 **44px**，宽度为弹窗 **80%**，圆角 **8px**
- 可点击：主题色背景 + 白色文字，有按压效果
- 不可点击：浅灰背景 + 深灰文字，无点击反馈
- Toast、消息样式与全局一致

## 4. 交互流程

### 4.1 发起申请

1. 用户点击「添加好友」→ 调用服务端创建好友申请（HTTP）。
2. 成功：按钮变为「申请中」；Toast：**好友申请已发送**。
3. 失败（网络等）：按钮不变；Toast：**网络异常，请稍后重试**（PRD）。

**重复点击「申请中」**：不重复发送；Toast：**申请已发送，请等待对方处理**。

### 4.2 申请结果（发起人侧）

| 结果 | 按钮 | Toast |
|------|------|--------|
| 对方同意 | 已添加 | 你们已成为好友 |
| 对方拒绝 | 添加好友 | 好友申请被拒绝 |
| 申请过期（7 天未处理） | 添加好友 | 好友申请已过期 |

### 4.2.1 发起人侧状态同步（已定）

- **主路径**：依赖 **用户 WS** 下行（对方同意 / 拒绝 / 申请过期），更新战绩弹窗按钮与 §4.2 所列 Toast。
- **对齐**：打开**对手战绩弹窗**、**应用回到前台** 时 **HTTP 拉取**与某用户的「关系 + 申请状态」（见 **§7.1**、**§5.3**），避免仅依赖 WS 时漏更新。

### 4.3 被申请人侧（本期：WS + 弹框，无消息中心页）

**本期不做独立「消息中心」列表页**（见 §10）。

- 收到好友申请时，依赖 **用户级 WebSocket** 下发；客户端弹出系统级或游戏内弹框：**{玩家昵称} 请求添加你为好友**，操作：**同意 / 拒绝**。
- 操作后双方状态一致；打开战绩弹窗时状态与全局一致（以服务端为准）；必要时配合 **HTTP 拉取** 兜底（见 §5.3）。

### 4.4 双向同时申请

- 若 A→B、B→A **两条待处理申请同时存在**：**任意一方在弹框中点击「同意」**，即建立好友关系，两条申请按服务端规则一并关闭。
- **双方同时**可能收到「对方请求加你为好友」类弹框提醒（与单向申请一致，由 WS 推送触发）；UI 需避免重复阻塞（如同期只展示一个弹框队列或合并文案，实现细节由客户端约定）。

## 5. 实时通道：用户级 WebSocket（已定）

### 5.1 问题

对局使用现有房间 WebSocket（如 `/ws/gomoku`）。**离开房间后连接断开**，若好友推送仅依赖房间 WS，会**收不到**后续通知。

### 5.2 方案

- **单独增加用户级 WebSocket 端点**（例如 `/ws/user` 或 `/ws/notify`），用 **session / 用户身份** 建连，**不绑定 roomId**。
- **房间 WS**：继续负责行棋、房间内聊天等。
- **用户 WS**：推送好友申请到达、同意/拒绝、过期同步等。

语义上等同于「单独 topic」；实现为**第二条 `wx.connectSocket` + 独立 Handler**，而非在房间 WS 内用 `type` 硬塞（避免离房即失联）。

### 5.3 HTTP 兜底

用户 WS 也会因杀进程、断网、退后台而断开，因此：

- 打开**对手战绩弹窗**、**应用回到前台 / 重连用户 WS 后**等时机，应用 **HTTP 拉取**当前好友关系与待处理申请状态，与「状态全局一致」对齐。

### 5.4 双 WebSocket 工程约定（已定，采用推荐方案）

与对局 **房间 WS**（`/ws/gomoku`）并列，新增 **用户 WS** 专用端点（推荐路径 **`/ws/user`**，名称以最终实现为准）。

| 维度 | 房间 WS | 用户 WS |
|------|---------|---------|
| 绑定 | **房间 `roomId`** + 对局/聊天 | **用户 `userId`**，不绑房间 |
| 建连 | 进房 | **登录成功拿到 session 后**建立 |
| 断连 | **离房 / 终局离开房间** | **仅退出登录 / 明确登出**；切房、回首页、结束对局 **不断** |
| 消息 | 行棋、RESET、房间聊天等 | 好友申请、同意/拒绝结果、关系与申请状态同步等 |

**鉴权**：与房间 WS **同一套 session/token 校验**（query 或首包 `auth`，与现有实现对齐）。

**心跳与重连**：两条连接 **各自**维护心跳（如 20–30s 量级）与断线重连（指数退避 + 上限；`onShow`/网络恢复时优先尝试一次）。**不要**用「房间已连接」推断「用户 WS 已连接」。

**多实例**：用户 WS 推送需能到达任意节点上的连接（与房间 WS 一致：**Redis Pub/Sub、MQ 等**按需选用）。

**微信侧限制**：实现前核对当前文档对 **并发 WebSocket 条数** 的限制。若 **仅允许一条连接**，则退化为 **单 WS + 消息内 `channel`/`type` 区分房间业务与用户业务**（本期优先双端点，遇限再改）。

## 6. 业务规则

- 建立好友关系：需**至少一方**在收到申请后的弹框中选择「同意」；若双方互发两条待处理申请，**任一方一次同意**即建立关系并关闭相关申请（见 §4.4）。
- 申请有效期 **7 天**，超时自动失效。
- **防骚扰**：同一用户对同一对手，**24 小时内最多发起 3 次**；超限按钮置灰，提示：**申请过于频繁，请 24 小时后再试**（服务端强校验）。  
  **计数规则（已定）**：仅当本次 `POST` 产生 **新有效申请**（响应 `status: CREATED`）计 **1 次**。**幂等**返回 `PENDING`（重复提交）、**已是好友** `ALREADY_FRIENDS`、以及其它不产生新申请的响应 **不计入** 该 3 次。
- **人机 / 假对手**：**不隐藏**「添加好友」按钮，展示与真人对战一致；若对手为服务端人机账号，是否允许建立游戏内好友以**服务端校验与错误提示**为准（前端不因人机隐藏入口）。

### 6.1 发起申请接口幂等（已定）

**服务端**

- 对「同一对 `(fromUserId, toUserId)` + 待处理 **PENDING**」做唯一约束或等价业务规则：**不允许多条并行 pending**。
- 若已存在 pending，再次 `POST` 发起申请：**不新建第二条**；返回 **200**，body 带当前状态（如 `status: PENDING`）及 `friendRequestId`（若有）。
- 若双方**已是好友**：返回成功语义下的 **`ALREADY_FRIENDS`**（或等价字段），前端展示「已添加」。
- 可选：请求头 **`Idempotency-Key`**（UUID）便于客户端重试；无则依赖上两条亦可避免脏数据。

**客户端**

- 「申请中」按钮不可点（见 §3、§8）；请求进行中用 **`addFriendInFlight`**（或等价）忽略重复点击。
- 收到 **200 + PENDING**（重复提交）：保持「申请中」，**不**再次 Toast「好友申请已发送」（与 PRD「申请已发送，请等待对方处理」区分场景：仅用户误触已禁按钮时可弱提示）。

**响应体约定（示例）**

```json
{
  "status": "CREATED | PENDING | ALREADY_FRIENDS | RATE_LIMITED",
  "friendRequestId": "string"
}
```

（字段名以实现为准，`RATE_LIMITED` 对应 §6 频控。）

## 7. HTTP 与 WebSocket 契约（草案，随实现可微调）

### 7.1 REST 接口（草案）

| 方法 | 路径（示例） | 说明 |
|------|----------------|------|
| `POST` | `/api/social/friend-requests` | 发起申请；body：`targetUserId`；响应见 §6.1 |
| `POST` | `/api/social/friend-requests/{id}/accept` | 同意（被申请人）；幂等：已关闭则返回当前态 |
| `POST` | `/api/social/friend-requests/{id}/reject` | 拒绝（被申请人） |
| `GET` | `/api/social/friend-status?userId=` | 与指定用户的按钮态：是否好友、pending 方向、`friendRequestId` 等（打开战绩弹窗、回前台时拉取） |

**约定**

- **同意 / 拒绝 / 查询** 一律走 **HTTP**；**不**用 WS 回写业务结果（WS 仅用于 **通知与状态广播**）。
- 鉴权：与其它 API 相同 **session / Bearer**（与现有 `roomApi` 一致）。
- 错误码（示例）：`401` 未登录；`403` 非本人可操作；`404` 申请不存在或已失效；`409` 状态冲突（可选）；频控 **`429`** 或 body `status: RATE_LIMITED`（与 §6.1 统一）。

### 7.2 用户 WS 下行消息（草案）

连接：**`/ws/user`**（见 §5.4）。每条建议为 JSON，含 **`type`** 与 **`payload`**（字段名可统一为驼峰）。

| `type` | 含义 | payload 要点 |
|--------|------|----------------|
| `FRIEND_REQUEST_INCOMING` | 有新的好友申请待处理 | `friendRequestId`, `fromUserId`, `fromNickname`（展示用） |
| `FRIEND_REQUEST_RESOLVED` | 某条申请已关闭（同意/拒绝/过期/互申合并关闭） | `friendRequestId`, `outcome`: `ACCEPTED` \| `REJECTED` \| `EXPIRED`, `counterpartUserId` |
| `FRIENDSHIP_UPDATED` | 好友关系变化（可选，与上一类可合并实现） | `peerUserId`, `nowFriends`: boolean |

**说明**：发起人侧 §4.2 的 Toast（对方同意/拒绝/过期）可由 `FRIEND_REQUEST_RESOLVED` + `outcome` 驱动；被申请人弹框由 `FRIEND_REQUEST_INCOMING` 驱动。若服务端合并事件类型，需在实现说明里固定一种枚举，与客户端共用常量。

### 7.3 与房间 WS 的边界

- **`/ws/gomoku`**：仅对局与房间聊天；**不**承载上述 `FRIEND_*` 类型（避免离房丢消息）。

| 重复点「申请中」 | 不重复发送；Toast：申请已发送，请等待对方处理 |

## 9. 验收标准（PRD）

- 三种按钮态正确切换。
- 发送、同意、拒绝、过期流程正确。
- 24h/3 次限制生效，且与 **§6 频控计数规则**（仅 `CREATED` 计次）一致。
- **用户 WS** 与 **§7** 契约联调通过；发起人侧 **§4.2.1** 同步路径（WS + HTTP）无状态错乱。
- **幂等**（§6.1）重复提交不产生多余数据。
- 无 UI 错位、遮挡；交互无明显卡顿。
- 不影响对局结束页及对局中其他原有功能。
- **历史战绩列表**路径的加好友 **不在本期验收范围**（见 **§10**、**§13**）。
